# S3 定时调度（Schedule）测试计划

## 1. 测试目标与判定原则

负责人：`yulin`。本模块验证 GitCode Actions 的 `schedule.cron` 调度、UTC 时区、默认分支限制、最小间隔、延迟以及 cron 字段和扩展语法。

所有 24 个 TC 的预期均以 Excel 为准。调度配置被平台接受不等于 cron 语义已经验证；除纯配置边界用例外，必须观察到与计划时间匹配的真实 Schedule Run，才能判定 `PASS`。

执行前先评审单个用例的目标 cron、UTC 计划时间、观察窗口、所需分支、允许的延迟和证据来源。未审查或未记录计划时间的 cron 不得启用。

## 2. 范围

`TC-237`、`TC-427`、`TC-428`、`TC-429`、`TC-430`、`TC-471`、`TC-472`、`TC-473`、`TC-474`、`TC-475`、`TC-476`、`TC-477`、`TC-478`、`TC-479`、`TC-505`、`TC-506`、`TC-507`、`TC-508`、`TC-509`、`TC-510`、`TC-511`、`TC-512`、`TC-562`、`TC-563`。

每个 TC 使用独立 `TC-<编号>-<name>.yml` 文件。提交到仓库的本地版本默认只保留 `workflow_dispatch`；实际 `on.schedule` 块仅在单个受控测试窗口内临时启用，禁止同时启用两个测试 cron。

## 3. 用例分组与评审要求

| 用例组 | TC | Excel 验证重点 | 执行前必须确定 |
| --- | --- | --- | --- |
| 基础调度与延迟 | TC-237、TC-427、TC-430、TC-563 | cron 到点触发、UTC 换算、可接受分钟级延迟 | UTC 计划时间、发布时间、观察结束时间、允许延迟 |
| 默认分支限制与间隔 | TC-428、TC-562、TC-429 | 非默认分支不调度；小于五分钟间隔无效或受限 | 默认分支名称、对照分支、固定无运行观察窗口、最小间隔判定口径 |
| 标准 cron 运算符 | TC-471、TC-472、TC-473、TC-474、TC-505、TC-506、TC-507、TC-512 | `*`、`,`、`-`、`/` 和范围加步长的真实触发行为 | 每个表达式对应的目标分钟/小时、避免与其他 cron 重叠的时间窗 |
| 字段边界 | TC-475、TC-476、TC-477、TC-478、TC-479 | 分、时、日、月、星期字段的合法范围 | Excel 指定的有效字段值、日历限制、是否需真实运行 |
| 非标准扩展 | TC-508、TC-509、TC-510、TC-511 | `?`、`L`、`W`、`#` 的接受或执行语义 | 该 TC 是配置接受测试还是需要等待日期语义，最长观察周期 |

## 4. 单次执行流程

1. 选择唯一 TC，复核其 Excel 预期和本计划的评审条目。
2. 在本地记录 `planned_utc`、提交 SHA、目标分支、目标 cron、观察窗口与结束时间。
3. 仅为该文件临时增加 `on.schedule`，推送到正确分支后开始观察。
4. 通过 Actions Run API 记录 Run 创建时间、事件类型、Job 开始时间及 Runner UTC 时间；三者不可混用。
5. 对要求“不应触发”的用例，观察窗口结束前不得提前下结论。
6. 结束后移除 `on.schedule`，通过 Contents API 或仓库文件检查确认已清除，再继续下一个用例。

### 4.1 逐 TC cron、观察窗口和事件归属

所有“应触发”用例以 GitCode Actions Run 的事件类型为 `schedule`、Run 创建时间落在 `planned_utc` 加允许延迟范围内、且 Job 实际启动为三项共同证据。`push`、`workflow_dispatch`、手动重跑或其他事件创建的 Run 一律不能作为 schedule 通过证据。发布后的第一个理论触发时刻为观察起点；每次观察都在 `notes.md` 写入精确的 `planned_utc`、`window_end_utc` 和实际 Run 创建时间。

| TC | Excel cron/配置 | 预期与可执行断言 | 最小观察规则 |
| --- | --- | --- | --- |
| TC-237 | `0 2 * * *` | 默认分支上每天 02:00 UTC 创建 `schedule` Run | 从下一个 02:00 UTC 起，观察至 02:15 UTC；运行即记录延迟 |
| TC-427 | `0 2 * * *` | cron 使用 UTC；本地展示时间必须换算为 02:00 UTC | 选择能清楚换算的本地时区，记录本地理论时刻和 UTC 时刻；按 TC-237 窗口观察 |
| TC-428 | 在非默认分支定义 `schedule` | 非默认分支不创建 `schedule` Run | 覆盖一个完整理论触发时刻及 15 分钟延迟窗口；同时保留默认分支正向对照成功证据 |
| TC-429 | 边界探针 `*/1 * * * *` | 低于 5 分钟不生效或受平台限制 | 至少覆盖 6 个理论分钟和 5 分钟延迟窗口；仅记录 `schedule` Run 数量与时刻 |
| TC-430 | `0 2 * * *` | 可存在数分钟调度延迟，不要求秒级精确 | 与 TC-237 同次或独立运行均可；记录计划时刻与 Run 创建时刻差值 |
| TC-471 | `* * * * *` | 每分钟触发 | 连续观察 6 个理论分钟和额外 5 分钟延迟窗口；记录每个 `schedule` Run |
| TC-472 | `0 0 * * 1,3,5` | 周一、周三、周五 00:00 UTC 触发 | 在未来目标星期运行；至少观察目标时刻加 15 分钟。不可等待时保留配置证据并标记 `PARTIAL` |
| TC-473 | `0 0 * * 1-5` | 周一至周五 00:00 UTC 触发 | 在目标工作日运行；观察目标时刻加 15 分钟 |
| TC-474 | `*/15 * * * *` | 每 15 分钟触发 | 从已知整 15 分钟边界起观察两个相邻触发点，每点加 5 分钟延迟窗口 |
| TC-475 | `30 * * * *` | 分钟字段 `30` 在 0-59 的合法范围内 | 观察下一个整点 30 分及其后 15 分钟 |
| TC-476 | `0 2 * * *` | 小时字段 `2` 以 UTC 表示，属于 0-23 的合法范围 | 按 TC-237 观察，并记录 UTC 字段值 |
| TC-477 | `0 0 1 * *` | 日期字段 `1` 在 1-31 的合法范围内 | 下月 1 日 00:00 UTC 加 15 分钟；周期过长时仅记录配置接受并标记 `PARTIAL` |
| TC-478 | `0 0 1 1 *` | 月字段 `1` 在 1-12 的合法范围内 | 下一个 1 月 1 日窗口；周期过长时仅记录配置接受并标记 `PARTIAL` |
| TC-479 | `0 0 * * 0` | 星期字段 `0` 表示周日 | 下一个周日 00:00 UTC 加 15 分钟 |
| TC-505 | `5,15,25 * * * *` | 每小时第 5、15、25 分钟触发 | 从下一小时的第 5 分钟起，连续观察三个理论触发点，各加 5 分钟延迟窗口 |
| TC-506 | `0-5 * * * *` | 每小时第 0 至 5 分钟连续触发 | 从下一整点起观察 6 个理论分钟及额外 5 分钟延迟窗口 |
| TC-507 | `*/15 * * * *` | 每 15 分钟触发 | 与 TC-474 相同，观察两个相邻触发点 |
| TC-508 | `0 2 ? * *` | `?` 不指定日或周，日和周不能同时指定 | 先保存配置接受/拒绝结果；若接受，等待下一 02:00 UTC 加 15 分钟。平台拒绝即为 Excel 预期反例 |
| TC-509 | `0 0 L * *` | 每月最后一天 00:00 UTC 触发 | 保存配置结果；等待下一个月末加 15 分钟，周期不足时为 `PARTIAL` |
| TC-510 | `0 0 15W * *` | 每月 15 日最近工作日 00:00 UTC 触发 | 计算下一次 15 日的最近工作日，观察该时刻加 15 分钟；周期不足时为 `PARTIAL` |
| TC-511 | `0 0 * * 2#3` | 每月第三个周二 00:00 UTC 触发 | 计算下一第三个周二，观察该时刻加 15 分钟；周期不足时为 `PARTIAL` |
| TC-512 | `5-45/10 * * * *` | 每小时第 5、15、25、35、45 分钟触发 | 从下一小时第 5 分钟起，观察五个理论触发点，各加 5 分钟延迟窗口 |
| TC-562 | 在非默认分支定义 `schedule` | 仅默认分支生效，非默认分支不得触发 | 与 TC-428 相同；必须已有默认分支正向对照，否则为 `BLOCKED` |
| TC-563 | 任意已验证有效 cron | 调度允许分钟级延迟，不保证秒级精确 | 使用 TC-237 或其他有效 cron，记录计划与创建时间差；延迟本身不是失败 |

### 4.2 TC-429 的 Excel 源数据矛盾

TC-429 的预期写明“低于 5 分钟不生效”，但 Excel YAML 示例为 `*/5 * * * *`，该表达式恰为 5 分钟，不属于“低于 5 分钟”。因此正式边界探针使用 `*/1 * * * *`；`*/5 * * * *` 只能作为五分钟可触发的正向对照，不能用于判定 TC-429。执行记录必须同时保存这一源数据矛盾和最终采用 `*/1` 的理由。

## 5. 结果判定

- `PASS`：真实调度行为符合 Excel 预期；仅配置用例则需有明确的接受/拒绝证据。
- `FAIL`：平台接受无效边界、拒绝 Excel 规定的有效形式，或在非目标时间/分支发生调度。
- `PARTIAL`：配置已接受但因日期依赖或观察窗口不足，无法验证完整语义。
- `BLOCKED`：基础调度不可用，导致无法将结果归因于具体 cron 语法。

## 6. 本地证据和安全要求

每个 TC 在 `evidence/TC-<编号>/` 保留计划时间、YAML、Run/Job API 响应、观察记录和清理证据。任何时候主分支或测试分支上最多存在一个启用的 S3 测试 cron；测试会话关闭前必须不存在启用的 yulin S3 cron。

## 7. 已发现的平台问题与规避规则

1. 在 `main` 临时启用有效 cron `15 5 19 7 *` 后，18 分钟观察窗口未产生匹配的 schedule Run。基础调度未工作前，不能继续把运算符、范围、边界或扩展表达式的配置接受写成 `PASS`。
2. 基础 scheduler 不触发时，涉及 UTC、延迟、默认分支、最小间隔、运算符与字段语义的用例均应记录 `BLOCKED`，因为无法将结果归因给单个 cron 表达式。
3. 每次 cron 观察结束后必须立即从 `main` 删除该文件，并通过 Contents API 确认删除；在基础 scheduler 恢复前不得重新启用多个 S3 cron。
4. `jobs=[]` 的 Workflow 配置失败不能提供任何 schedule 行为证据。
