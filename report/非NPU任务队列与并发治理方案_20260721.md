# 非 NPU 任务队列与并发治理方案

**日期**：2026-07-21
**状态**：方案，未获得正式写入授权
**适用范围**：Karmada 正式环境中路由到 `001` 的非 NPU Volcano Job，优先覆盖 `argo/ascend-*` 批量 CI 任务。

## 1. 目标与结论

当前 COP、p20 与 Namespace propagation 策略已经实现两个目标：非 NPU Job 不会进入 NPU 节点，并优先选择 pre-paid 节点。它们不能控制 Job 到达速率。

本方案解决的是瞬时批量提交导致的大量 Pending 与 post-paid ECS 弹出，而不是改变既有调度落点策略：

```text
批量 CI 提交
  -> CI 提交限流（主控制点，任务尚未创建）
  -> Karmada / Volcano 调度
  -> Volcano Queue 资源额度（第二道护栏）
  -> pre-paid 优先；容量不足时才回退 post-paid
```

**推荐决策**：先在 CI/工作流入口实现按架构分组的 Job 并发限制；Volcano Queue 资源额度作为后续防御性护栏，必须先在测试环境验证不会让等待 Job 反向触发 Cluster Autoscaler 扩容。

本方案不修改：

- COP `non-npu-vcjob-node-affinity`。
- p20、p10、`has-cpu` 或 `dispatch/auto` 标签。
- HNA 和 Cluster Autoscaler 参数。
- 现有业务 Pod、节点或 ECS。

## 2. 观测依据

数据来自 `2026-07-20 20:45` 至 `2026-07-21 09:47` 的 15 份小时完整快照和 79 条十分钟轻量指标，采集过程仅使用 Kubernetes `get/list`。

| 观察项 | 结果 |
| --- | --- |
| 策略字段 | COP、p20、11 条 Namespace CPP 全程无漂移 |
| 采样 Pod 落点 | 14/14 非 NPU Pod 位于 pre-paid 节点 |
| 最大活动 Job | 125 |
| 最大 Pending Pod | 27 |
| post-paid 范围 | 3 至 40 台 |
| 22:08 扩容 | post-paid 3 -> 20，活跃 Job 80、Pending 27 |
| 22:18 扩容 | post-paid 20 -> 31 |
| 00:10 扩容 | post-paid 22 -> 40 |

因此问题是瞬时到达速率超过可承接 pre-paid 容量，而非 COP 没有优先选择 pre-paid。

## 3. 实际候选容量

最新正式只读核对显示，`001` 有 21 台 Ready pre-paid 节点，但不能把它们作为一个统一容量池：

| 节点类别 | 数量 | 对普通非 NPU Job 的含义 |
| --- | ---: | --- |
| pre-paid 且带 `accelerator/huawei-npu` | 11 | 被 COP 的 `DoesNotExist` 硬约束排除 |
| pre-paid 且非 NPU | 10 | 进入候选集合 |
| 非 NPU 且 amd64 | 6 | 仅可承接 amd64 Job |
| amd64 且带 `cache=true:NoSchedule` | 2 | 无对应 toleration 的普通 Job 不可使用 |
| 普通 amd64 非 NPU候选 | 约 4 | amd64 Job 的主要候选池 |
| arm64 非 NPU候选 | 约 4 | arm64 Job 的主要候选池 |

所以并发控制必须按架构分组；一个全局“最多 10 个 Job”的限制不能反映 amd64 与 arm64 各自独立的小候选池。

## 4. 主方案：CI/工作流提交限流

### 4.1 控制位置

在创建 Volcano Job 的上游 CI/工作流调度器中控制。后续等待任务不创建 Karmada Job、ResourceBinding、成员 Job 或 Pod。

这与“只在 Volcano Queue 中等待”的根本区别是：未创建的任务不会形成 Pending Pod，因此不会被 Cluster Autoscaler 当作新增容量需求。

具体实现落点按实际平台选择：

| 任务入口 | 实现方式 |
| --- | --- |
| GitHub Actions | `concurrency`、matrix `max-parallel`，或外层派发器 |
| Jenkins | Throttle Concurrent Builds、Lockable Resources 或共享队列 |
| Argo Workflows | `spec.parallelism` 或 Semaphore |
| 自研 CI | 按架构维护 worker semaphore / 消费者并发数 |
| 批量脚本 | 在 `kubectl apply` 前使用有界 worker pool，完成后再放行下一项 |

### 4.2 队列模型

为每个架构建立独立逻辑队列：

```text
non-npu-amd64
non-npu-arm64
```

队列键至少包含：

```text
业务类别 + 架构 + 目标环境
```

不得将 NPU Job 与非 NPU Job 放入同一个限流预算，避免一个类别耗尽另一个类别的并发额度。

### 4.3 初始灰度参数

没有完整的每 Job 资源规格历史时，不应直接全量把并发设为 5。建议先采用保守、可观察的灰度：

| 队列 | 初始运行中 Job 上限 | 单次放行 | 观察周期 |
| --- | ---: | ---: | --- |
| `non-npu-amd64` | 4 | 1 | 至少两个 22:00/00:00 高峰 |
| `non-npu-arm64` | 4 | 1 | 至少两个 22:00/00:00 高峰 |

`4` 不是容量上限结论，而是基于当前每类约 4 台普通候选节点的安全灰度起点。每次有一个 Job 进入终态后，最多放行同架构的一个等待 Job。

若单个 Job 是多 Pod 或 gang Job，额度必须按 Job 的总 Pod requests 重新折算；不能只按 Job 数量套用上表。

### 4.4 放行规则

1. 上游收到任务后先读取其架构：`kubernetes.io/arch=amd64`、`kunpeng-arm=true` 或明确的镜像/模板标签。
2. 获取对应队列令牌后才创建 Karmada Volcano Job。
3. Job 进入 `Completed`、`Failed`、`Terminated` 或 `Aborted` 后释放令牌。
4. 创建失败、ResourceBinding 未 Scheduled 或新 Work 首次 Apply 失败时立即释放令牌，并按现有失败流程告警。
5. 未声明架构的任务不得自动进入任一队列；应失败并要求任务模板声明架构，避免调度到错误容量池。

## 5. 第二道护栏：Volcano Queue 资源额度

Volcano Queue 可以限制同一 Queue 同时使用的 CPU/内存资源，但不应把它当作“最多 N 个 Job”的精确工具。不同 Job 的规格不同，历史中存在约 `31 CPU / 80Gi`、`64 CPU / 128Gi`、`90 CPU / 120Gi` 等差异。

建议的目标形态是按架构或业务类别分 Queue，并以 requests 配额限制并发资源，例如：

```yaml
# 仅示意；实际 CRD 字段、队列名称和额度须以 live API 为准。
apiVersion: scheduling.volcano.sh/v1beta1
kind: Queue
metadata:
  name: non-npu-amd64
spec:
  capability:
    cpu: "<approved-cpu-budget>"
    memory: "<approved-memory-budget>"
```

正式前置验证必须回答：超过 Queue capability 的 Job 是被 Volcano 保持等待、拒绝，还是仍创建 Pending Pod；以及 Cluster Autoscaler 是否会对这些 Pod 扩容。若仍会扩容，Queue 不能替代上游提交限流。

## 6. 测试与灰度步骤

### 阶段 A：只读基线

1. 连续收集至少两个完整 24 小时周期。
2. 将每个非 NPU Job 按架构、总 CPU/内存 requests、replica/gang 数、创建时间、开始时间、终态时间记录下来。
3. 为高峰 Pending Pod 保留 `FailedScheduling`、Volcano PodGroup 与 volume/affinity Event 原因。
4. 从 CTS/CCE 获取实例 ID、任务 ID、节点池与创建结果，去重建立 ECS 扩容审计。

### 阶段 B：测试环境语义验证

1. 复用生产近似的 CPU/内存和 amd64/arm64 NodeSelector。
2. 验证上游 semaphore 在额度用尽时不创建 Karmada Job。
3. 验证 Volcano Queue capability 超额时的准确行为。
4. 观察 Autoscaler 对 Queue 等待 Pod 是否扩容。
5. 验证任务失败、取消、超时都会释放令牌，避免队列永久阻塞。

### 阶段 C：正式小范围灰度

1. 仅选择一个可回退、非关键 CI 流水线。
2. 先只启用 CI 侧架构队列，不改 Volcano Queue 和 HNA/Autoscaler。
3. 从 `amd64=4`、`arm64=4` 起步，持续两个高峰。
4. 用现有 10 分钟采集对比灰度前后：峰值 Pending、post-paid 峰值、任务端到端时延、失败率和 HNA/ECS 动作。
5. 达到门槛后再逐项调整并发或增加 Queue 护栏；每次只改一个变量。

## 7. 成功、停止与回滚门槛

### 成功标准

需由业务和成本负责人补齐具体数值。建议至少同时满足：

| 指标 | 建议定义 |
| --- | --- |
| Pending SLO | 例如 p95 等待时间不超过 `<T>` 分钟 |
| 任务时延 | 相比基线的 p95 端到端时延增幅不超过 `<X>%` |
| post-paid 峰值 | 相比同类高峰基线下降，且不通过提高任务失败率实现 |
| 调度正确性 | 新非 NPU Pod 持续满足非 NPU required 与 pre-paid 偏好 |
| 可靠性 | 新 Work 首次 Apply 成功率不下降，ResourceBinding 无异常目标 |

### 立即停止灰度

- 关键业务超过已批准的 Pending SLO。
- 任务失败率、取消率或 Work 首次 Apply 失败率显著升高。
- 新非 NPU Pod 违反 NPU 硬排除，或出现非预期集群目标。
- 队列令牌泄漏导致任务长时间无法放行。

### 回滚方式

CI 队列方案的回滚是将受影响流水线并发限制恢复为原值或禁用 semaphore。它不修改已创建 Job，不删除 Pod，不驱逐节点，也不改变 COP、Queue、HNA 或 Autoscaler。

## 8. 需确认的决策项

在任何正式写入或灰度前，必须由相关负责人明确：

1. amd64 与 arm64 任务的最大可接受等待时间。
2. 每次 22:00/00:00 高峰允许的 post-paid ECS 峰值或成本上限。
3. 哪个 CI/工作流入口拥有实际 Job 创建控制权。
4. Job 是否允许在 CI 平台等待，还是必须立即创建 Kubernetes Job。
5. 是否可以为 `cache=true:NoSchedule` 节点增加业务 toleration；该问题须由节点所有者确认，禁止为扩容优化直接移除污点。
6. Volcano Queue 实际 CRD 的 capability/queue 配置及其与 Cluster Autoscaler 的交互测试结果。

## 9. 明确不建议的做法

- 不通过修改 COP、p20 或 Cluster 标签实现排队。
- 不为了利用 11 台 NPU pre-paid 节点而移除 `accelerator/huawei-npu DoesNotExist` 硬约束。
- 不先提高 HNA 阈值来压制扩容；这可能只把成本问题转成更长 Pending。
- 不把“10 个 Job”作为永久参数；应根据架构、每 Job requests 和 gang 规模持续调整。
- 不用 HNA `status.conditions` 条数估算 ECS 创建量；高峰时其历史会快速被覆盖。
