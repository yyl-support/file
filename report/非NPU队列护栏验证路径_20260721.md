# 非 NPU Queue 护栏验证路径

**日期**：2026-07-21
**状态**：测试验证方案；无正式写入授权
**唯一优化目标**：尽可能减少批量非 NPU Job 导致的 post-paid ECS 瞬时弹出，同时维持已批准的任务等待 SLO。

## 1. 背景与问题定义

现有调度优化已验证有效：COP 保留 NPU 节点硬排除并对 pre-paid 节点给予 weight 100 软偏好；p20 和 11 条 Namespace CPP 没有漂移。

但 `2026-07-20 20:45` 至 `2026-07-21 09:47` 观测到：

| 时刻 | post-paid 节点变化 | 活跃 Job | Pending Pod |
| --- | --- | ---: | ---: |
| 22:08 | 3 -> 20，+17 | 80 | 27 |
| 22:18 | 20 -> 31，+11 | 38 | 0 |
| 00:10 | 22 -> 40，+18 | 64 | 0 |

问题不是 COP 未优先使用 pre-paid，而是瞬时任务到达速率超过适配非 NPU pre-paid 候选池承载能力。非 NPU Pod 随后合法回退到 post-paid，Cluster Autoscaler/HNA 扩容链路快速补充 ECS。

## 2. 验证假设

建立非 NPU 专用 Volcano Queue 并限定 CPU/内存 capability 后，超过已批准资源预算的非 NPU Job 将被 Queue 限制，从而减少瞬时可调度需求和 post-paid ECS 弹出。

该假设只有在以下条件成立时才有效：

```text
Queue 超额 Job
  -> 不会形成可触发 Cluster Autoscaler 的大规模 Pending Pod
  -> 或 Autoscaler 明确忽略由 Queue capability 阻塞的 Job
```

若超额 Job 仍形成普通 Pending Pod，且 Cluster Autoscaler 仍将其视为新增容量需求，则 Queue capability 不能减少 ECS 弹出。此时 Queue 只是一层业务资源护栏，主控制必须回到 CI 创建前限流。

## 3. 设计原则

1. 不修改现有 COP、p20、p10、Cluster 标签、HNA 或 Cluster Autoscaler。
2. 不修改当前 `shared-flexible-queue`：它由 Karmada 管理，且包含 NPU 与非 NPU 业务，不能作为非 NPU 专属护栏。
3. 不在成员集群直接修改 Karmada 管理对象；Queue 必须从 Karmada 控制面创建并通过现有 Queue propagation 链路同步。
4. 不手工删除节点、ECS、Job 或业务 Pod。
5. 先验证语义，后决定 capability 数值；不能仅凭“限制为 5 或 10 个 Job”设置正式配额。
6. amd64 与 arm64 必须独立观察，因为二者的非 NPU pre-paid 候选节点池不同。

## 4. 测试目标架构

```text
Karmada 控制面
  -> non-npu-guardrail-queue Queue
  -> 现有 volcano-global-all-queue-propagation
  -> 测试成员集群 Queue

Karmada 非 NPU Volcano Job
  -> 测试用 Override/策略自动改写 spec.queue
  -> non-npu-guardrail-queue
  -> Volcano Queue capability 判定
```

Queue 从 Karmada 创建，是因为正式 `shared-flexible-queue` 已由 `volcano-global-all-queue-propagation` 管理并同步到成员集群。成员侧直接修改会被 Karmada Work 覆盖。

## 5. 分阶段验证

### 阶段 A：测试环境只读基线

1. 读取测试环境 Volcano `Queue` CRD 的 `spec.capability`、`status.allocated` 和 `status.state`。
2. 读取测试 Cluster Autoscaler/HNA 配置与当前日志入口。
3. 确认可用的非 NPU amd64/arm64 节点数量、requests 容量、污点和架构约束。
4. 记录现有共享 Queue、Job 模板、ResourceBinding、Work 和 PodGroup 的基线。
5. 不创建或修改任何资源。

### 阶段 B：Queue 传播验证

1. 在测试 Karmada 创建唯一测试 Queue：`non-npu-guardrail-queue`。
2. 仅使用最小必要字段：`priority`、`weight`、`reclaimable`、`capability`。
3. 通过现有全局 Queue propagation 传播至测试成员集群。
4. 验证：Karmada Queue、CRB、成员 Work 和成员 Queue 均成功，且目标 Queue `state=Open`。
5. 验证不影响 `shared-flexible-queue`、NPU Job 或其他 Queue。

### 阶段 C：非 NPU Job 自动导流验证

1. 创建只匹配测试非 NPU Volcano Job 的 Karmada Override。
2. 仅改写新测试 Job 的 `spec.queue` 为 `non-npu-guardrail-queue`。
3. 创建一个低资源 amd64 测试 Job 和一个低资源 arm64 测试 Job。
4. 验证 ResourceBinding、成员 Work、成员 Job 和 Pod 的 queue/affinity/节点架构。
5. 清理测试 Job；不修改生产策略对象。

**关键约束**：先确认 `spec.queue` 的 override 不会尝试改写存量 Volcano Job。Volcano Job 的部分字段不可变，任何存量协调失败都必须与新 Job 的首次 Apply 结果分开判断。

### 阶段 D：超额 Queue 语义验证

1. 将 Queue capability 设为足以承接一批测试 Job、但不足以承接第二批的固定 CPU/内存预算。
2. 先提交预算内 Job，确认 `status.allocated` 与预期资源一致。
3. 再提交超额 Job，保持全部资源、架构和 Job 模板相同，仅使 aggregate requests 超出 capability。
4. 连续采集至少 30 分钟：
   - Queue `status.allocated` 与 `state`。
   - Job phase、Pod phase、PodGroup phase 与 Event 原因。
   - ResourceBinding 与 Work 首次 Apply 状态。
   - Cluster Autoscaler/HNA Event、节点池目标与节点总数。
   - post-paid 节点数和节点注册时间。
5. 完成首批 Job 后，验证超额 Job 是否自动获得资源、开始运行，并使 allocated 正确释放/再分配。

### 阶段 E：与无护栏基线对比

使用同规格、同架构、同数量 Job，在隔离测试窗口分别运行：

| 对比组 | Queue 行为 | 目的 |
| --- | --- | --- |
| 基线组 | 现有共享 Queue，无专用 capability | 记录 Pending 与 Autoscaler 基线 |
| 护栏组 | 非 NPU 专用 Queue，有 capability | 验证 Queue 是否减少扩容需求 |

不可同时运行两组，避免相互竞争资源并污染结论。

## 6. 必须采集的证据

每个阶段至少保存以下只读证据：

```text
时间戳
Karmada Queue / CRB / Work JSON
成员 Queue / Job / Pod / PodGroup JSON
Queue allocated、state、capability
Job/Pod/PodGroup Event
成员节点：名称、注册时间、计费模式、架构、Ready
HNA action history 与 Event
Cluster Autoscaler 日志或可用 Event
```

ECS 成功创建最终以 CCE/CTS 的实例 ID、任务 ID、节点池和状态为准。节点数量变化和 HNA action 只能作为线索，不能代替 ECS 审计。

## 7. 验收标准

### Queue 语义通过

必须同时满足：

1. 超额 Job 的阻塞原因明确可识别为 Queue 资源限制。
2. 超额 Job 不会导致超出测试基线的 post-paid 节点扩容。
3. 首批 Job 终态后，等待 Job 能自动继续，且资源记账正确。
4. 新 Job 的 ResourceBinding、Work 首次 Apply、required NPU 排除和 pre-paid 偏好保持正确。
5. 现有共享 Queue、NPU Job 和非测试业务无行为变化。

### 不通过的判定

出现任一情况即判定 Queue 不能作为减少 ECS 弹出的主控制：

- 超额 Job 仍让 Cluster Autoscaler/HNA 扩容，且扩容幅度与无护栏基线相近。
- 超额 Job 的 Pod 被作为普通 `Unschedulable` 容量需求处理。
- Queue 限制影响共享 Queue、NPU Job 或无关业务。
- Queue allocated 不释放，或等待 Job 不会在资源归还后恢复。
- Override 对存量 Job 造成实际业务失败或新 Work 首次 Apply 失败。

## 8. 正式灰度门禁

满足以下全部条件后，才可以单独申请正式小范围灰度：

1. 阶段 A 至 E 的原始证据与结论通过评审。
2. 明确 Pending SLO：等待 Job 可接受的最大等待时长、p95 等待时长和允许数量。
3. 明确成本边界：同类高峰允许的 post-paid 节点峰值或预算。
4. 明确覆盖范围：仅一个可回退、低风险业务 Namespace/流水线。
5. 明确 Queue capability、Override 匹配范围、回滚对象和负责人。
6. 已确认生产 Karmada Queue propagation 不会将测试 Queue 误传播到不应覆盖的集群。
7. 任何正式写入均获得单独批准；不得与 COP、HNA、Autoscaler 或标签改动合并执行。

## 9. 正式灰度成功标准

与同类高峰基线比较，至少观察两个高峰/回收周期：

| 指标 | 目标 |
| --- | --- |
| 十分钟 post-paid 节点增量 | 下降 |
| post-paid 节点峰值 | 下降或在成本上限内 |
| Pending 等待时长 | 不超过已批准 SLO |
| Job/Work 成功率 | 不下降 |
| 新非 NPU Pod 落点 | 持续满足 NPU 排除，保持 pre-paid 优先 |
| 非目标业务 | 无 Queue、调度或可用性回归 |

## 10. 回滚

测试或正式灰度发生停止条件时：

1. 停止创建新的灰度测试 Job。
2. 移除只匹配灰度 Job 的 queue override。
3. 恢复该小范围 Job 的原始 Queue 行为。
4. 待灰度 Job 自然完成；不驱逐、不删除正在运行的业务 Pod。
5. 在确认无引用后，从 Karmada 删除专用 Queue，并验证 CRB/Work/成员 Queue 已按控制面收敛。

不得通过删除 post-paid 节点、修改 HNA/Autoscaler 或去除 NPU 硬排除作为回滚方式。

## 11. 结论

Queue 护栏值得验证，因为它有机会把非 NPU 突发需求限制在 pre-paid 可承接预算内，从而减少 post-paid ECS 的瞬时弹出。

但它不是已验证的解决方案。只有证明“Queue 超额等待不会继续触发 Autoscaler 扩容”后，才可以将其作为正式减少 ECS 弹出的控制手段；否则必须在 CI 创建 Job 前进行限流。
