# 正式环境调度策略自动采集脚本使用文档

更新时间：2026-07-20 CST

## 1. 目的

本工具在 3 至 4 天内持续采集正式 Karmada 调度状态，以形成可审计证据，验证当前非 NPU Job 调度策略是否持续有效。

验证目标：

- COP 持续保留 `accelerator/huawei-npu DoesNotExist` required 排除。
- COP 持续保留 `node.cce.io/billing-mode In [pre-paid]`、weight 100 的软偏好。
- p20 `non-npu-vcjob-prefer-001` 持续使用 `has-cpu=true`，且当前精确解析为 `001`。
- 11 条 Namespace CPP 持续使用 `dispatch/auto=true`，且当前精确解析为 `001/wlcb`。
- `001` 和 `wlcb` 持续 Ready。
- 自然产生的非 NPU Volcano Pod 实际优先落在 pre-paid、非 NPU 节点；若出现 post-paid 落点，保留其发生时的状态供容量回退分析。

## 2. 安全边界

采集脚本仅使用 Kubernetes `get/list` 请求，不会执行以下操作：

- 不会创建、更新、Patch 或删除任何 Kubernetes 资源。
- 不会修改 COP、CPP、Cluster 标签、Queue、HNA 或 Cluster Autoscaler。
- 不会删除 post-paid 节点或驱逐业务 Pod。
- 不会运行 `workflow.sh`。

采集和报告数据均写入本地已忽略的 `runtime/` 目录，不会影响版本库内容。

## 3. 文件说明

```text
60-policy-observation.sh                 单次正式环境只读快照采集
61-policy-observation-report.sh          生成最近 3 或 4 天 Markdown 报告
62-policy-light-metrics.sh                单次十分钟轻量指标采集
lib/policy_observation.py                快照归纳和报告逻辑
systemd/                                  Linux systemd service、timer 和环境文件示例
schedule-policy-observation.plist.example macOS launchd 每小时执行示例
schedule-policy-light-metrics.plist.example macOS launchd 每十分钟执行示例
runtime/policy-observation/              本地快照、日志和报告输出目录
```

默认 kubeconfig 继承自 `config.sh`：

```text
Karmada: /Users/gorden/huawei/code/karmada/正式/正式贵阳karmada.yaml
001:     /Users/gorden/huawei/code/karmada/正式/0001.yaml
wlcb:    /Users/gorden/huawei/code/karmada/正式/wlcb.yaml
```

如需使用其他只读 kubeconfig，可在启动命令前覆盖 `KARMADA_KUBECONFIG`、`CLUSTER_001_KUBECONFIG` 和 `CLUSTER_WLCB_KUBECONFIG`。

## 4. 采集内容

每小时完整采集均保存原始 Kubernetes JSON 和 `summary.json` 汇总，范围如下：

| 来源 | 采集对象 | 用途 |
| --- | --- | --- |
| Karmada | Cluster、COP、p20 CPP、全部 CPP | 策略字段、标签目标集、集群 Ready 校验 |
| Karmada | 非 NPU Volcano Job、Work、ResourceBinding | Job/Work 状态和控制面调度证据 |
| Karmada | Warning Event | 保留控制面异常线索 |
| 001/wlcb | Node | Ready、计费模式、NPU 标签 |
| 001/wlcb | Pod | 将非 NPU Volcano Pod 对应到实际节点计费模式 |
| 001/wlcb | HNA、Warning Event | 保留 HNA 状态、HNA 相关 Event 和成员集群异常线索 |

采用分层采集：每小时一次完整快照，另每 10 分钟一次轻量指标。3 天约产生 72 个完整快照和 432 条轻量记录，4 天约产生 96 个完整快照和 576 条轻量记录；对控制面和本机开销很低。

小时级快照用于策略字段、实际 Pod 落点和 HNA 原始状态复核。十分钟轻量记录仅保存非 NPU Job/Work、未调度 Pod、节点计费模式、HNA conditions 与 HNA Event 聚合指标，用于捕获短高峰和节点数变化。因此报告会区分“策略完整性证据”“运行时 Pod 落点证据”和“弹性扩容线索”，不会在没有自然业务样本时作出过强结论。

## 5. 前置检查

在脚本目录执行：

```bash
cd /Users/gorden/huawei/code/karmada/正式/调度优化自动化
command -v kubectl
command -v python3
./60-policy-observation.sh --once
./62-policy-light-metrics.sh --once
```

最后一条命令会执行一次只读采集，并输出类似路径：

```text
runtime/policy-observation/snapshots/20260720-170000/summary.json
```

若该步骤失败，不应启动定时采集。优先检查 kubeconfig 路径、当前凭据是否具备对应资源的 `get/list` 权限，以及网络连通性。

## 6. Linux 启动分层定时采集

生产持续采集建议使用 Linux 常开主机和 `systemd`。以下单元均为 `Type=oneshot`，只会运行既有只读采集脚本；不会获得任何 Kubernetes 写入能力。

### 6.1 部署前准备

将整个 `调度优化自动化` 目录部署到 Linux 主机的固定路径，例如 `/opt/karmada-scheduling-automation`。安装 `kubectl`、`python3`，创建运行账号和数据目录：

```bash
sudo useradd --system --home-dir /nonexistent --shell /usr/sbin/nologin karmada-observer
sudo install -d -o karmada-observer -g karmada-observer -m 0750 \
  /var/lib/karmada-policy-observation
sudo install -d -o root -g karmada-observer -m 0750 \
  /etc/karmada-policy-observation
```

将三个只读 kubeconfig 放入 `/etc/karmada-policy-observation/`，命名为 `karmada.yaml`、`001.yaml`、`wlcb.yaml`。kubeconfig 必须仅对 root 和 `karmada-observer` 可读：

```bash
sudo chown root:karmada-observer /etc/karmada-policy-observation/*.yaml
sudo chmod 0640 /etc/karmada-policy-observation/*.yaml
```

创建运行环境文件：

```bash
sudo cp /opt/karmada-scheduling-automation/systemd/karmada-policy-observation.env.example \
  /etc/karmada-policy-observation.env
sudo chown root:root /etc/karmada-policy-observation.env
sudo chmod 0600 /etc/karmada-policy-observation.env
sudoedit /etc/karmada-policy-observation.env
```

确认 `KUBECTL_BIN` 是 Linux 主机上的绝对路径。默认 kubeconfig 和数据目录可保留示例值；如使用其他路径，必须同步修改该环境文件。

### 6.2 安装并启用 timer

安装 unit 文件、重新加载 systemd，并立即执行一次采集验证：

```bash
sudo cp /opt/karmada-scheduling-automation/systemd/karmada-policy-*.service /etc/systemd/system/
sudo cp /opt/karmada-scheduling-automation/systemd/karmada-policy-*.timer /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl start karmada-policy-observation.service
sudo systemctl start karmada-policy-light-metrics.service
```

确认两次手工采集均成功后，启动并设为开机自动启用：

```bash
sudo systemctl enable --now karmada-policy-observation.timer
sudo systemctl enable --now karmada-policy-light-metrics.timer
systemctl list-timers 'karmada-policy-*'
```

- `karmada-policy-observation.timer`：每小时完整采集，最多附加 2 分钟随机延迟。
- `karmada-policy-light-metrics.timer`：首次启动后约 2 分钟开始，每次成功完成后间隔 10 分钟采集轻量指标，最多附加 30 秒随机延迟。
- `Persistent=true`：主机停机期间错过的 timer 会在系统恢复后补跑一次。补跑只能获取恢复时的当前状态，不能补回停机期间结束的 Job、短暂 Pending 或瞬时 HNA 动作。

查看执行情况和日志：

```bash
systemctl status karmada-policy-observation.timer karmada-policy-light-metrics.timer
systemctl status karmada-policy-observation.service karmada-policy-light-metrics.service
journalctl -u karmada-policy-observation.service -u karmada-policy-light-metrics.service --since '1 hour ago'
```

Linux 输出目录为：

```text
/var/lib/karmada-policy-observation/snapshots/
/var/lib/karmada-policy-observation/light-metrics/
/var/lib/karmada-policy-observation/reports/
```

## 7. macOS 启动分层定时采集

macOS 可继续使用 `launchd`，但休眠期间不会按时执行；它不适合作为本次 3 至 4 天连续采集的首选宿主。脚本不会自行注册系统级定时任务，需要人工执行以下命令：

```bash
cd /Users/gorden/huawei/code/karmada/正式/调度优化自动化
mkdir -p runtime/policy-observation
cp schedule-policy-observation.plist.example \
  ~/Library/LaunchAgents/io.karmada.production-policy-observation.plist
cp schedule-policy-light-metrics.plist.example \
  ~/Library/LaunchAgents/io.karmada.production-policy-light-metrics.plist
launchctl bootstrap "gui/$(id -u)" \
  ~/Library/LaunchAgents/io.karmada.production-policy-observation.plist
launchctl bootstrap "gui/$(id -u)" \
  ~/Library/LaunchAgents/io.karmada.production-policy-light-metrics.plist
```

确认任务已加载：

```bash
launchctl print "gui/$(id -u)/io.karmada.production-policy-observation"
launchctl print "gui/$(id -u)/io.karmada.production-policy-light-metrics"
```

`io.karmada.production-policy-observation` 的 `StartInterval=3600` 表示每小时完整采集一次。`io.karmada.production-policy-light-metrics` 的 `StartInterval=600` 表示每 10 分钟采集一次轻量指标。任务执行日志写入：

```text
runtime/policy-observation/launchd.out.log
runtime/policy-observation/launchd.err.log
runtime/policy-observation/light-metrics.launchd.out.log
runtime/policy-observation/light-metrics.launchd.err.log
```

首次加载后可立即手工执行一次 `./60-policy-observation.sh --once`，无需等待下一个小时周期。

## 8. 查看采集状态

查看已完成快照：

```bash
ls -lt runtime/policy-observation/snapshots
```

查看最新快照摘要：

```bash
latest="$(ls -dt runtime/policy-observation/snapshots/* | head -n 1)"
python3 -m json.tool "${latest}/summary.json"
```

每个快照目录包含 `summary.json` 及采集时的原始对象 JSON。原始数据用于在报告发现策略漂移、集群不 Ready、post-paid 落点或异常事件时进行复核。

查看当天轻量指标：

```bash
less runtime/policy-observation/light-metrics/metrics-$(date '+%Y%m%d').jsonl
```

轻量数据按天追加到 JSONL 文件，每行是一个独立 JSON 记录；不保存 Pod、Job 的全量对象。

## 9. 生成持续验证报告

连续采集满 3 或 4 天后，在脚本目录执行：

```bash
./61-policy-observation-report.sh --days 3
./61-policy-observation-report.sh --days 4
```

Linux 使用 systemd 默认运行目录时，执行报告命令需要显式使用相同目录：

```bash
cd /opt/karmada-scheduling-automation
sudo RUNTIME_DIR=/var/lib/karmada-policy-observation \
  ./61-policy-observation-report.sh --days 3
```

报告默认生成在：

```text
runtime/policy-observation/reports/scheduling-policy-3d-<timestamp>.md
runtime/policy-observation/reports/scheduling-policy-4d-<timestamp>.md
```

报告包括：

- 快照数、最早和最晚采集时间。
- 所有策略完整性和集群健康条件均满足的快照数。
- 有活动非 NPU Job 的快照数。
- 已观测非 NPU Pod 的 pre-paid/post-paid 节点计费分布。
- 十分钟粒度的活动非 NPU Job 峰值、未调度 Pod 峰值和计费节点数量范围。
- HNA conditions、HNA Event 线索和连续轻量记录识别到的节点数上升时间点。
- 与证据强度相匹配的结论和局限说明。

## 10. 报告判读

可支持“当前策略有效”的充分证据为：

1. 统计窗口内每个快照均满足策略完整性和 `001/wlcb Ready` 条件。
2. 统计窗口内存在自然产生的非 NPU Job/Pod 样本。
3. 观测到的非 NPU Pod 均落在 pre-paid、非 NPU 节点，或 post-paid 落点均能结合当时 pre-paid 容量不足得到合理解释。

以下情况不能直接否定策略：

- 出现 post-paid Pod。COP 的计费模式规则是 soft preference；符合 NPU required 排除的 pre-paid 容量不足时，调度到 post-paid 是允许的容量回退。
- 某些小时没有活动非 NPU Job。此时可证明策略未漂移，但不能单独证明运行时偏好被业务触发。
- HNA conditions 条数较少。HNA 仅保留有限历史，不能用其数量估算累计扩容次数。
- 节点数量上升。它是 ECS 弹出的观测线索，最终必须按实例 ID、任务 ID、节点池和 CTS/CCE 审计去重确认成功 ECS 数量。

以下情况应停止使用“持续有效”结论并复核原始快照：

- `has-cpu=true` 不再精确等于 `001`。
- `dispatch/auto=true` 不再精确等于 `001/wlcb`。
- COP 缺少 NPU required 排除或 pre-paid 权重 100 偏好。
- p20 或任一 Namespace CPP 选择器偏离预期。
- `001` 或 `wlcb` 出现非 Ready 状态。
- 非 NPU Pod 落在 NPU 节点。

## 11. 停止采集与保留数据

Linux 停止和禁用 timer：

```bash
sudo systemctl disable --now karmada-policy-observation.timer
sudo systemctl disable --now karmada-policy-light-metrics.timer
```

macOS 完成 3 至 4 天观察后，停止 `launchd` 任务：

```bash
launchctl bootout "gui/$(id -u)" \
  ~/Library/LaunchAgents/io.karmada.production-policy-observation.plist
launchctl bootout "gui/$(id -u)" \
  ~/Library/LaunchAgents/io.karmada.production-policy-light-metrics.plist
```

该操作只会停止本机计划任务，不会删除快照、报告或正式环境中的任何资源。保留 `runtime/policy-observation/` 中的全部数据，直至报告评审和必要的异常复盘完成。

## 12. 常见排障

| 现象 | 排查方式 |
| --- | --- |
| `kubectl not found` | 安装 kubectl，或设置 `KUBECTL_BIN` 为可执行文件绝对路径。 |
| `Forbidden` | 核对对应 kubeconfig 是否有 Cluster、COP、CPP、Job、Work、ResourceBinding、Node、Pod、Event 的只读权限。 |
| 某个集群请求超时 | 检查 VPN/网络、kubeconfig server 地址及 API Server 健康；不要因采集失败修改生产对象。 |
| 没有生成报告 | 确认已至少采集一个位于指定最近 3 或 4 天窗口内的 `summary.json`。 |
| 没有 Pod 落点证据 | 延长自然业务采样窗口；不要为获取证据在正式环境主动创建压测任务。 |
| `launchd` 未执行 | 用 `launchctl print` 检查两个任务状态，并查看各自的 `launchd.err.log`。 |
| systemd timer 未执行 | 使用 `systemctl list-timers 'karmada-policy-*'` 检查下次执行时间，用 `journalctl -u <service>` 查看失败原因。 |
| Linux 服务无法读取 kubeconfig | 核对 `/etc/karmada-policy-observation/` 的路径、属主、组和 `0640` 权限；运行账号为 `karmada-observer`。 |
| Linux 主机停机或网络中断 | `Persistent=true` 会在恢复后补跑一次，但不能补回停机期间的短时负载证据；将中断时间记录在最终报告中。 |

## 13. 操作约束

1. 正式环境默认只读；本工具不构成任何策略、标签、HNA 或 Autoscaler 写入授权。
2. 不直接运行完整 `workflow.sh`。
3. 不手工删除 post-paid 节点，不驱逐业务 Pod。
4. 不为了改善报告数据而改动 `has-cpu` 或 `dispatch/auto` 标签。
5. 生成结论前，先确认采样窗口覆盖至少一个有自然非 NPU 业务负载的时段。
